# Build For One Research Workflows

## What Problem This Solves

Scholars often imagine they must build general AI tools for everyone. That is the wrong starting point.

Start by building one workflow that solves one repeated problem in your own research.

## Examples

- a conference tracker for your subfield
- a paper-reading workflow for asset pricing
- a replication-package checker for RA work
- a Stata debugging skill for your empirical projects
- a tough-referee project for one paper draft
- a seminar Q&A assistant for one presentation
- a data dictionary reviewer for one project

## Recommended Workflow

1. Identify one repeated task.
2. Write the exact inputs you usually have.
3. Write the output you actually need.
4. Add verification steps.
5. Test on a small nonconfidential example.
6. Turn it into a skill, checklist, or project instruction only after it works.

## What Not To Do

- start with a generic tool for all scholars
- automate before understanding the task
- share a workflow before testing failure cases
- use sensitive data in early experiments
