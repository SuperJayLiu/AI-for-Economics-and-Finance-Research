# Replication Project

## Purpose

Help understand, audit, and reproduce an economics or finance paper's empirical workflow.

## Best For

- replication packages
- codebase orientation
- table-to-script mapping
- variable construction review
- identifying missing documentation
- creating replication checklists

## Not Suitable For

- claiming replication success without running code
- replacing manual inspection of data and outputs
- bypassing data-use restrictions

## What To Upload

- README files
- code scripts
- data dictionaries
- public sample data if allowed
- table shells
- replication notes

## What Not To Upload

- restricted datasets
- licensed datasets that cannot be redistributed
- private author correspondence

## Project Instructions

- Map scripts to tables and figures.
- Identify required data inputs.
- Flag missing files, undocumented steps, and fragile assumptions.
- Do not claim code works unless outputs are provided.
- Separate code structure from empirical interpretation.

## Recommended Workflow

1. Inventory files.
2. Identify the master script or workflow order.
3. Map scripts to tables and figures.
4. Inspect variable construction.
5. Run code locally when permitted.
6. Compare produced outputs to paper tables.
7. Document gaps and unresolved differences.

## What The AI Should Produce

- file inventory
- execution order draft
- table-to-code map
- variable construction notes
- replication checklist
- missing-information list

## What The Scholar Must Verify

- code execution
- data access rules
- package versions
- random seeds
- table output
- sample construction
- any claimed discrepancy

## Common Mistakes

- assuming code order from filenames alone
- overlooking data-use constraints
- trusting AI's code reading without running scripts
- failing to record software versions

## Disclosure Notes

Keep a replication log with tool use, files inspected, commands run, outputs produced, and unresolved issues.
