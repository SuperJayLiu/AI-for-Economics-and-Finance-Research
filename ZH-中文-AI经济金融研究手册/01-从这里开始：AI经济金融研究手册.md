# 01 从这里开始：AI 经济金融研究手册

这一页是中文读者的核心手册。它解释本仓库为什么存在、AI 能帮什么、不能信什么、怎样保护数据、怎样用 Git 和 AI-use log 让研究可追踪。

> [!IMPORTANT]
> AI 可以降低研究劳动成本，但不能让研究问题自动变重要，不能让识别策略自动可信，不能让引用自动真实，也不能替研究者承担责任。

## 快速判断：我应该怎么开始？

| 你的情况 | 建议路径 |
| --- | --- |
| 刚开始了解 AI | 先读本页，再用一个公开、低风险任务测试一个 skill。 |
| 已经有一个研究想法 | 用研究想法压力测试模板，先判断问题、机制、数据和识别风险。 |
| 正在写文献综述 | 用文献矩阵和来源支持表，不要让 AI 自己发明文献。 |
| 正在写实证方法 | 用经济学或金融学实证方法模板，然后逐项对照代码和表格。 |
| 想让 AI 改代码或改文件 | 先设置 Git、`.gitignore`、`AGENTS.md`、`AI-USE-LOG.md`。 |
| 想用 agent 自动化 | 先要求 AI 给计划、风险、可编辑文件、禁止编辑文件和验证命令。 |

## 把模糊请求改成可核查工作流

| 模糊请求 | 更好的 AI 工作流 |
| --- | --- |
| “帮我总结文献。” | “只使用我提供的 12 篇论文，按 research question、data、identification/model、finding、limitation、与我项目关系建表。缺失信息标为 [需要核查]。” |
| “帮我写 methods。” | “只使用下面已核查事实，写 data、sample、equation、identification assumptions、inference、limitations，并把缺失信息标为 [需要作者补充]。” |
| “帮我清理数据。” | “先检查文件夹并提出 raw-to-derived pipeline。不要改文件。列出 raw data、derived data、code、outputs、隐私风险和需要我批准的检查。” |
| “帮我做 slides。” | “根据论文做 25 分钟 seminar outline。每页说明 claim、evidence、需要的图表、过度声称风险和 speaker note。” |
| “帮我用 agent。” | “先给计划、可读文件、可改文件、禁止文件、验证命令和人工批准点，不要直接执行。” |

不确定怎么写任务时，可以复制：

```text
我想把 AI 用在这个经济学/金融学研究任务上：
[描述任务]

请先帮我把它变成可控制、可核查的工作流。必要时先提澄清问题。然后说明：
1. 安全输入；
2. 禁止输入；
3. 预期输出；
4. AI 可以做什么；
5. AI 不能替我决定什么；
6. 我应该如何核查；
7. 需要在 AI-use log 里记录什么。
```

## 最小安全配置

一个严肃研究项目至少需要：

| 文件或工具 | 为什么需要 |
| --- | --- |
| `README.md` | 说明项目目的、数据、代码、论文结构。 |
| `DATA.md` | 记录数据来源、权限、敏感性和能否上传 AI 工具。 |
| `AGENTS.md` 或 `CLAUDE.md` | 告诉 AI agent 什么能做、什么不能做。 |
| `AI-USE-LOG.md` | 记录 AI 做了什么、人类检查了什么、还不确定什么。 |
| `.gitignore` | 防止原始、受限或私人数据被提交到 GitHub。 |
| Git/GitHub | 查看差异、恢复旧版本、分支实验、审查 AI 修改。 |

## 初学者工具词汇表

| 工具/概念 | 简单解释 | 适合做什么 | 主要风险 |
| --- | --- | --- | --- |
| AI | 泛指能生成、分类、预测或执行任务的系统。 | 语言、代码、整理、解释。 | 输出不是 evidence。 |
| LLM | 大语言模型，根据上下文生成文字和代码。 | 解释代码、总结你提供的材料、起草清单。 | 不是数据库，可能 hallucinate。 |
| ChatGPT | OpenAI 的聊天产品，Projects 可保存聊天、文件和指令。 | 一个论文、文献综述或展示建一个 Project。 | 注意隐私设置和项目指令。 |
| Claude | Anthropic 的聊天产品，适合长文档和项目式工作。 | 读论文、批改草稿、整理材料。 | 仍需核查来源。 |
| Codex | OpenAI 的代码和 repo agent。 | 修改代码、运行检查、看 Git diff。 | 必须使用 Git 并审查每个修改。 |
| Claude Code | Claude 的命令行/repo agent。 | 在项目文件夹里执行写作、代码和 agent 工作流。 | 权限和 auto mode 需要谨慎。 |
| VS Code | 代码编辑器，可看文件、运行终端、管理 Git。 | 写代码、调试、看 diff。 | 不能替代可复现性。 |
| MCP/connector | 让 AI 连接外部工具或数据源。 | Zotero、GitHub、搜索、文件和数据库。 | 权限、token 成本和数据暴露风险。 |

## 这些技术词在研究项目里是什么意思？

| 术语 | 研究场景中的例子 |
| --- | --- |
| repo | 一个论文、复现包或数据项目的文件夹加版本历史。 |
| commit | 一次保存的项目快照，例如“修正 Table 2 代码后的版本”。 |
| diff | AI 修改前后具体改了哪些行。 |
| branch | 一个安全的旁支版本，例如 `try-new-event-study`，可以让 AI 试验而不影响主项目。 |
| worktree | 同一个项目的第二个文件夹，用来让另一个 agent 在不同 branch 上并行工作。 |
| `.gitignore` | Git 的“不要上传/不要追踪清单”，例如写入 `data/raw/` 后，原始数据不会被 Git 追踪。 |
| raw data | 原始数据，按收到时的样子保存，不直接编辑。 |
| derived data | 由代码从 raw data 生成的清洗或合并数据。 |
| `AGENTS.md` | 给 coding agent 的说明书，告诉它哪些文件能读、能改、不能动，以及要跑什么检查。 |
| `CLAUDE.md` | Claude Code 读取的项目记忆文件，记录项目阶段、规则、命令和注意事项。 |
| MCP/connector | 让 AI 连接 GitHub、Zotero、Drive 或数据库的“接口”。 |
| skill | 可重复使用的研究流程，例如“检查 DiD 设计”或“设计 WRDS 合并方案”。 |
| AI-use log | 记录 AI 用途的研究日志：工具、任务、文件、采纳内容、核查和不确定性。 |

如果 AI 回答中出现看不懂的技术词，可以直接问：

```text
请用普通语言解释每个技术词，并为每个词给一个经济学或金融学研究中的例子。
```

## AI 在研究中的合适位置

AI 比较适合：

- 整理已有文献笔记；
- 把模糊研究想法拆成问题、机制、数据和方法；
- 解释 Stata、R、Python、LaTeX 代码；
- 起草清洗、合并、变量构造、EDA、表格和图形代码；
- 设计 text-as-data 和 LLM measurement 的验证流程；
- 辅助 structural/quantitative model 的 moments、parameters、counterfactual 和 welfare 检查；
- 起草实证方法段落，但不能替你决定识别策略；
- 检查表格、图、文字和代码是否一致；
- 练习研讨会报告、求职报告和审稿回复；
- 创建检查清单、日志、代码仓库结构和项目指令。

AI 不应该替你做：

- 最终研究问题选择；
- 最终识别策略判断；
- 因果解释；
- 最终文献贡献判断；
- 审稿判断；
- 上传受限数据、保密材料或审稿材料的决定；
- 最终投稿或政策/投资建议。

## 核查本身是一项技能

不要只说“人工核查”。要说明怎么核查。

| AI 输出 | 应该怎么核查 |
| --- | --- |
| 引用或文献结论 | 查 DOI、期刊页、原文页码，并确认原文支持这句话。 |
| 代码 | 用小型 toy data 先跑出已知答案，再跑真实数据。 |
| 数据合并 | 每步打印匹配率、重复 key、未匹配记录和时间规则。 |
| 系数解释 | 检查单位、log/percent 解释、baseline、置信区间和研究设计。 |
| 理论推导 | 检查符号、假设、limiting cases、维度和反例。 |
| LLM 生成变量 | 保存 prompt、模型、日期、设置、输出，并做人类标签验证和敏感性分析。 |

简短规则：

```text
如果我不能说清楚如何核查这个 AI 输出，就不应该把它用于研究。
```

## 成熟度阶梯

| 等级 | 名称 | 典型做法 | 主要风险 |
| --- | --- | --- | --- |
| 1 | 随手聊天 | 临时问 ChatGPT/Claude 一个问题 | 输出听起来对但可能错 |
| 2 | 结构化聊天 | 给背景、要求不确定性、人工核查 | 难以复现 |
| 3 | AI 项目 | 一个论文或任务一个 ChatGPT/Claude Project | 项目指令太弱 |
| 4 | GitHub 辅助工作流 | AI 在 repo 中改代码、改文字 | 改错文件、覆盖文件 |
| 5 | skills 工作流 | 重复任务变成可复用 skill | skill 没有验证步骤 |
| 6 | agent 工作流 | AI 计划、改文件、运行代码、检查输出 | 权限、隐私、过度自动化 |
| 7 | 研究操作系统 | AI + GitHub + 日志 + 数据规则 + 更新系统 | 维护成本高 |

不要在没有 Git、数据规则和人工验证习惯之前跳到 agent 自动化。

## 机构规则优先

本仓库提供工作流建议，不提供合规许可。使用 AI 前必须检查：

- 学校或雇主的 AI 使用规则；
- 研究伦理或 IRB 规则；
- 数据提供方 license；
- 期刊、会议、基金或课程规则；
- 合作者、导师、PI 的要求。

如果任何规则比本仓库更严格，遵守更严格的规则。如果不确定能否上传或处理某些材料，不要上传。

## 数据安全默认规则

| 材料 | 默认规则 |
| --- | --- |
| 公开宏观数据 | 通常可以，但仍要引用来源。 |
| 公开论文 PDF | 注意版权，不把 AI 摘要当事实。 |
| WRDS、CRSP、Compustat、Bloomberg、Refinitiv 数据 | 先检查 license，不要默认可上传。 |
| 银行、交易、家庭、公司、行政微观数据 | 不要上传到公开 AI 工具。 |
| 合作者论文草稿 | 先征得同意。 |
| 审稿材料 | 只有政策允许时才可使用 AI。 |
| 学生数据 | 避免使用公开 AI 工具。 |

更安全的替代方式：

- 用合成例子；
- 只提供变量字典，不提供记录；
- 在本地或机构批准工具中处理；
- 让 AI 写小型示例代码，再自己在本地运行。

## 研究者的核心习惯

```text
每次使用 AI 后，问五个问题：
1. 它用了什么输入？
2. 它产出了什么？
3. 哪些内容可能错？
4. 我人工核查了什么？
5. 需要记录到 AI-USE-LOG.md 吗？
```

## 好的 AI 使用习惯

| 习惯 | 做法 |
| --- | --- |
| 先澄清 | 如果任务、数据规则、目标读者或输出格式不清楚，先让 AI 提问。 |
| 解释术语 | 如果 AI 使用 `.gitignore`、branch、worktree、MCP、seed 等词，要求它用普通语言解释。 |
| 先做小测试 | 在真实数据前，先让 AI 构造 toy example，并给出已知答案。 |
| 分开劳动和判断 | AI 可以起草、整理、写代码和批评；研究者决定什么是真的、合规的、值得主张的。 |
| 以问题结尾 | 严肃任务的回答应列出还不确定的地方和研究者下一步要决定的问题。 |
